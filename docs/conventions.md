# 项目规范（conventions.md）

> AI 每次任务开始前必须阅读此文件。所有输出必须符合以下规范，风格不一致视为执行错误。

---

## 1. 命名规范

### 文件命名
- 全部小写，单词间用连字符：`task-001.md`、`report-task-001-2024-01-01.md`
- 任务文件：`task-{三位数编号}.md`，如 `task-001.md`
- 报告文件：`report-task-{编号}-{YYYY-MM-DD}.md`
- 决策文件：`adr-{三位数编号}-{简短标题}.md`

### 目录命名
- 全部小写，单词间用连字符
- 不允许使用空格或下划线

### 代码命名
- 变量/函数：snake_case（Python）或 camelCase（JS/TS），项目内统一
- 类名：PascalCase
- 常量：UPPER_SNAKE_CASE

---

## 2. 目录职责边界

| 目录 | 放什么 | 不放什么 |
|------|--------|----------|
| `src/` | 正式源代码、最终交付物 | 实验代码、草稿 |
| `docs/` | 稳定的参考文档 | 频繁变动的草稿 |
| `tasks/` | 任务定义文件 | 执行结果、代码 |
| `reports/` | 任务执行报告 | 源代码、数据文件 |
| `workspace/` | 临时上下文、中间产物、AI 草稿 | 最终交付物 |
| `tests/` | 测试代码 | 业务代码 |
| `logs/` | 运行日志 | 源代码 |

> ⚠️ workspace/ 是临时区。最终产物必须迁移至对应目录，不允许在 workspace/ 中长期存放交付物。

---

## 3. 文件粒度规范

- **一任务一文件**：每个任务独立一个 `task-{id}.md`，不合并多个任务到同一文件
- **一报告对应一任务**：每次任务执行完成后，新建对应报告文件
- **文档单一职责**：一个文档只描述一件事，超过 200 行考虑拆分

---

## 4. 任务格式规范

所有任务文件必须包含以下字段（参考 `tasks/task-template.md`）：

```
目标 / 输入 / 步骤 / 验收标准 / 风险 / 回滚方案 / 优先级 / 状态
```

- **验收标准必须是可执行命令或可断言的输出**，不接受"运行正常"、"功能完整"等模糊描述
- **回滚方案不可省略**，若无法回滚必须注明原因

---

## 5. 报告格式规范

所有报告必须包含（参考 `reports/task-001-report.md`）：

- 做了什么（行为描述，非结论）
- 修改内容（精确到文件和行号）
- 验证结果（附实际执行命令和输出，不允许只写"测试通过"）
- 风险点
- 下一步建议

---

## 6. AI 行为规范

- 每次任务结束必须更新 `STATE.md`
- 不允许在未经明确要求的情况下修改 `src/` 以外的正式文件
- 不允许删除任何 `reports/` 或 `tasks/` 中的历史文件
- 发现规范冲突时，优先遵循本文件，其次遵循 `AGENTS.md`，最后遵循任务文件

---

## 7. 任务拆解规则

> 从 PRD.md 自动拆解任务时，必须遵守以下规则。

### 粒度规则
- 一个任务最多对应**一天工作量**，超过则必须拆分
- 一个任务只做**一件事**，不允许把多个独立功能合并进同一任务
- 每个任务必须有**独立可验证的验收标准**，不依赖其他任务的完成才能验证

### 依赖规则
- 任务间依赖关系不超过**一层**（A→B 允许，A→B→C 必须重新设计）
- 依赖任务必须在当前任务的"输入"字段中明确标注

### 顺序规则
- 基础设施类任务（环境、数据、框架）必须排在功能类任务之前
- UI 类任务排在逻辑类任务之后
- 测试/验证类任务排在最后

### 命名规则
- 任务标题格式：`{动词}+{对象}`，如"实现用户登录"、"渲染控制图"
- 不允许用模糊标题，如"完成功能"、"处理数据"

### 数量规则
- 单个 Sprint（一周内）任务数量不超过 **7 个**
- 超过 7 个时，优先级低的任务移入下一个 Sprint

## 8. 黄金原则（Golden Principles）

> 这些原则防止 AI 复制次优模式，导致代码库长期熵增。
> 所有代码生成必须遵守，不允许以"快速完成"为由绕过。

### 工具复用优先
- 优先使用项目中已有的共享工具、函数、模块
- 禁止手写已存在的 helper 函数
- 发现重复代码时，必须提取为共享工具，而不是复制粘贴
- 引入新依赖前，先确认项目中是否已有等价实现

### 数据边界强制验证
- 数据必须在边界处验证（输入入口、模块接口、外部数据源）
- 禁止在内部函数中猜测数据结构或类型
- 禁止假设外部数据格式正确，必须有明确的验证或错误处理
- 数据结构变更必须同步更新 docs/data-schema/

### 最小修改原则
- 每次修改范围不超过任务要求
- 禁止在完成任务的同时顺手重构无关代码
- 发现需要重构的地方，记入 docs/technical-debt/，而非立即动手

### 可预测优于聪明
- 代码逻辑必须直白可读，禁止"聪明"的写法
- 注释说明"为什么"，而不是"做了什么"
- 复杂逻辑必须有对应的测试覆盖

---

## 9. 错误永久化规则（核心）

> 这是 Harness 的核心思想：每次出错，都要让同类错误永不再发生。
> 修复一个 bug 不够，必须把根本原因转化成永久约束。

### 强制流程
每次任务出错、阻塞或验收失败，修复完成后必须执行：

1. **识别根本原因**：不是"代码写错了"，而是"为什么会写错"
2. **转化成规则**：把根本原因写成一条可执行的约束
3. **写进永久文档**：
   - 命名/结构类问题 → 追加到 `docs/conventions.md`
   - AI 行为类问题 → 追加到 `AGENTS.md`
   - 环境/工具类问题 → 追加到 `docs/runbooks/README.md`
   - 数据格式类问题 → 追加到 `docs/data-schema/`
4. **在报告中记录**：注明"已永久化为规则：{规则内容}"

### 示例
```
❌ 错误处理：修复了 Plotly 版本不兼容的 bug
✅ 永久化处理：在 conventions.md 追加：
   "引入图表库前必须在 requirements.txt 锁定精确版本，
    不允许使用 >= 或 ~ 版本范围"
```

### 禁止行为
- 禁止修复 bug 后不做永久化
- 禁止把同一类错误修复两次而不产生规则

---

## 10. feature_list.json 维护规则

> feature_list.json 是功能状态的机器可读来源，优先级高于 tasks/README.md。

### 更新规则
- 功能完成后，只允许将 `passes` 字段从 `false` 改为 `true`
- 禁止修改已通过功能的 `steps` 字段
- 新增功能追加到数组末尾，不插入中间
- `id` 一旦分配不允许修改

### Generator 读取规则
每次任务开始前，Generator 必须：
1. 读取 `tasks/feature_list.json`
2. 找出所有 `passes: false` 的功能
3. 按 `priority` 排序，选择最高优先级的未完成功能
4. 确认对应的 `sprint` task 文件存在

### Evaluator 更新规则
验收通过后，Evaluator 负责将对应功能的 `passes` 改为 `true`，不允许 Generator 自己标记。
