# 阻塞反馈机制（blocked.md）

> 当 AI 执行任务遇到阻塞时，必须按此文件规定的流程处理。
> 阻塞不是失败，是信号——说明项目缺少某种能力、工具或文档。

---

## 核心原则

**AI 卡住时，不允许：**
- 反复尝试同一方法超过 3 次
- 静默跳过阻塞项继续执行
- 假设问题会自己消失

**AI 卡住时，必须：**
1. 立即停止当前任务
2. 识别阻塞原因（缺工具 / 缺文档 / 缺数据 / 需求不明确）
3. 记录到 `workspace/blocked-{task-id}-{date}.md`
4. 更新 STATE.md，标注任务为"已阻塞"
5. 输出阻塞报告，等待人工处理

---

## 阻塞分类与处理方式

| 阻塞类型 | 描述 | 处理方式 |
|----------|------|----------|
| 缺工具 | 需要某个能力但当前环境没有 | 记录所需工具，人工决定是否引入 |
| 缺文档 | docs/ 中没有足够的上下文 | 记录缺失信息，人工补充后重启任务 |
| 缺数据 | 需要数据但无法获取 | 记录数据需求，人工提供后重启 |
| 需求不明确 | 任务描述有歧义，无法判断正确方向 | 列出歧义点，人工澄清后重启 |
| 技术瓶颈 | 当前技术选型无法满足需求 | 记录到 technical-debt，提出替代方案 |

---

## 阻塞报告格式

文件路径：`workspace/blocked-{task-id}-{YYYY-MM-DD}.md`

```
# 阻塞报告：Task-{id}

## 阻塞时间
{YYYY-MM-DD HH:MM}

## 阻塞类型
缺工具 / 缺文档 / 缺数据 / 需求不明确 / 技术瓶颈

## 阻塞描述
（具体卡在哪一步，尝试了什么，为什么不行）

## 已尝试的方法
1.
2.
3.

## 需要人工提供的内容
（明确说明需要什么才能继续）

## 对项目的影响
（此阻塞是否影响其他任务）

## 建议的永久修复
（如何改进 docs/ 或工具，让下次不再遇到同样阻塞）
```

---

## 阻塞解除后

人工处理完成后，触发以下流程：

```
读取 workspace/blocked-{task-id}-{date}.md 和 STATE.md，
了解阻塞原因和处理结果，重新执行 tasks/task-{id}.md。
```

**同时必须：**
- 把"建议的永久修复"落实到对应的 docs/ 文件
- 把阻塞记录归档到 `docs/technical-debt/` 或 `docs/` 对应文档
- 这样下一次同类阻塞就不会再发生
